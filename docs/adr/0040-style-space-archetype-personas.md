# ADR 0040 — 风格空间 / 原型人格：把用户投影到 harness 态度锚点

## 背景

ccoach 的差异化价值不在「蒸馏一份高手技巧库」——技巧会过时、且脱离场景照搬就是 cargo cult。真正可迁移的是高手们对
**harness 工程本身的「态度」**：每个高手在 harness 上的取舍（要不要 plan mode、敢不敢 auto-accept、并不并行、投不投资配置/记忆、
每个 episode 跑不跑测试）都是一组 **revealed preference**，背后由其**典型场景 / 代码库性质 / 风险偏好**共同决定。

因此本 ADR 主张：ccoach 先把用户的真实行为数据**投影到一个「风格空间」**，先回答「**你是谁、你在什么场景**」，
再给「**对你这种人、在这种场景**」的建议——而不是发一套对所有人都一样的最佳实践。对**微操型**喊「放手」、
对**放养型**喊「多盯着」都是错的；脱离场景与现状风格的建议没有价值。

本切片**只定方向、不实现**（pending 蓝图占位）。它建立在三块已规划能力之上：

- E1/E2（[ADR 0032](0032-episode-abstraction-layer.md) / [ADR 0033](0033-episode-task-typing-within-type-normalization.md)）——
  `episode` 抽象与 `task_type` 分型，提供「一段连续工作」的边界与性质，是五轴指标的计算单元。
- E4（[ADR 0035](0035-incremental-cache-per-platform-profile.md)）——per-platform、增量缓存的 `profile`，
  是「现状风格向量」的落点与算分的输入面。

在这些底座落地前，本 ADR 不写代码，只固定原型集合、坐标轴定义、推断 vs 询问的边界、叙事/建议分层与对高手「浪费」的重定义。

## 决策

### D1 — 4–6 个近正交的锚点原型，用抽象原型名 + 真人「做出处」

定义一小组**风格锚点（archetype）**，每个锚点是风格空间里的一个坐标点。命名采用**抽象原型名**，
另以**真人作为「坐标出处」**（attribution，不进产品功能名）。三条理由：

- 不拿真人名做产品功能（人格标签直接挂真人名既不稳妥、也会被当作背书）；
- 原型的轴坐标要能**独立维护/校准**，与某个具体人解耦；
- 真人的工作流**会变**，锚点应是「某种态度」的抽象，而非某人当下的快照。

草案锚点（名称与出处均为草案，待校准）：

| 原型名（草案） | 态度内核 | 坐标出处（attribution） |
|---|---|---|
| 规划官型 | 前置规划、低配置高记忆、靠上下文复利 | Boris Cherny |
| 暴力循环型 | 大胆放养、靠循环与重试压平问题 | Geoffrey Huntley |
| 保守杠杆派 | 稳、强验证、谨慎放权 | Mitchell Hashimoto |
| 实证观察派 | 小步实证、边做边量、不轻信 | Simon Willison |
| 高吞吐 YOLO 派 | 高并行、高吞吐、坦然弃用 | Peter Steinberger |
| 改代码库不改 harness 派 | 几乎不投资 harness，改代码库适配 AI | Armin Ronacher |

原型数量控制在 4–6：太少不足以覆盖风格分布，太多会**互相靠近、丧失区分度**（见 OQ3）。锚点应尽量**近正交**。

### D2 — 五轴坐标（风格空间的维度）

风格空间由五个坐标轴张成；每轴由可从 episode/profile **派生**的指标定义（具体字段在实现期对齐 E1/E2/E4）：

1. **自主度**（微操 ↔ 放养）：打断率、auto-accept 占比。低 = 微操，高 = 放养。
2. **计划度**：plan mode 使用率、episode 的**「前轻后重」形状**（前期规划投入占 episode 的比重）。
3. **并行度**：同时活跃 session 数、跨 checkout / worktree 的程度。
4. **harness 投资度**：**拆成两子项**——`config_complexity`（配置复杂度）与 `memory_compounding`（记忆复利度）。
   二者必须分开，否则会误判：如规划官型**反而低配置、高记忆**，合成一轴会把它和「重配置」混为一谈。
5. **验证闭环度**：每 episode 的 test / typecheck 频次。

每个原型在这五轴上有一组目标坐标；用户向量也落在同一空间，便于求相似度与差距（见 D4）。

### D3 — 现状风格「推断不问」，目标风格「用户选」，错配时 skill 敢反对

区分两类风格，处理方式不同：

- **现状风格 = 推断、不问**。理由是 revealed preference：**自报严重失真**（人对自己的 harness 习惯描述与实际行为常背离），
  所以现状风格只从行为数据推断。且它是 **per-platform 且 per-project** 的——**同一个人**在公司大仓可能是「保守杠杆派」、
  周末 side project 可能是「YOLO 派」。不应给一个人贴一个全局标签（场景维度承接 [ADR 0037](0037-single-platform-default-compare-report.md)）。
- **目标风格 = 用户选**。意图（想往哪走）**数据测不出**，必须由用户表达。默认目标是**「当前风格的健康版」**——
  即**只修补当前风格的短板、不强行改变风格**；用户也可显式选一个原型作目标（如 `--coach-style boris`），
  此时给出**针对该目标的 gap analysis**。
- **目标-场景错配时，skill 敢反对**。当用户选的目标与其真实场景明显不匹配时（典型：用「暴力循环型」的胆量去跑「保守杠杆派」的场景，
  **又没有验证闭环**），skill 不应顺着用户，而应先指出风险（「先把验证 harness 建起来」）。**敢说「不」**正是教练区别于报表的地方。

### D4 — 叙事层与建议层分开（防止滑向星座式）

**绝不让人格标签直接生成建议**——「你是 X 型，所以应该……」会滑向星座/MBTI 式的话术，不可执行、无依据。两层职责切开：

- **建议层（structured，驱动建议）**：每个 archetype 蒸馏成**结构化数据**——五轴坐标 + 各指标的**目标区间** +
  每个指标对应的 **feature-first 解法**。建议的生成方式是：算**用户向量与目标向量在各指标上的 top-k 差距**，
  落到**具体指标 + 具体 feature**（而非落到人格名）。
- **叙事层（narrative，仅传播）**：人格名 / 相似度% / 代表人物，**只用于报告开头的故事**与
  **scorecard**（[ADR 0008](0008-gamified-shareable-scorecard.md)）的传播。叙事层不产出任何可执行建议。

### D5 — 重新定义高手的「浪费 token」

高手的行为会触发现有报告的「浪费」判定，必须重定义口径，否则**会把高手判成败家子**：例如规划官型/YOLO 派**并行跑很多会话**、
并**坦然接受 10–20% 的会话被废弃**——这是**调度成本，不是失败**。在风格空间语境下，「弃用率」应结合**并行度**与**原型期望**来解读：
处于高并行原型的用户，一定比例的会话弃用是其策略的正常组成部分，不应单独计为浪费。

## 后果

- ccoach 从「给所有人同一套最佳实践」升级为「**先定位、再因人/因场景施策**」，这是产品叙事的核心差异化点；建议从「通用清单」
  变成「你的向量 → 目标向量的 top-k 指标差距 → 具体 feature」，更可执行、更难被当成水文。
- 引入维护负担：原型坐标、各指标目标区间、feature 映射都是**需要持续校准的策展数据**（curated），不是纯算法产物（见 OQ1）。
- 强依赖 E1/E2/E4：五轴指标全部建立在 episode 抽象、任务分型与 per-platform profile 之上，本切片**必须**排在它们之后。
- 叙事/建议分层是硬约束：实现时人格标签与建议必须走**两条独立通路**，建议通路只认结构化指标，杜绝「标签→建议」直连。
- 隐私护栏不变：五轴指标均为 episode/profile 派生的**计数/比率/布尔**，遵守本仓既有红线（不读 assistant/思维链、不外发、写前脱敏截断）；
  可分享 scorecard 仍纯聚合、零 prompt 原文。现状风格是 per-machine/per-project 推断，不跨机器汇总。

## 开放问题

### OQ1 — 锚点轴坐标如何维护与校准

真人的工作流会变，锚点坐标与各指标目标区间需要一套**校准流程**（数据来源、更新频率、谁来策展、版本化）。
是否随版本固化一份带 changelog 的策展数据？如何避免坐标随某个人一时的风格漂移而抖动？

### OQ2 — 相似度算法

用户向量与原型向量的相似度怎么算（欧氏 / 余弦 / 各轴加权）、各轴是否同权、是否需要按轴做标准化（量纲差异大）、
相似度% 如何映射成对外可读的「像谁」叙事。

### OQ3 — 原型数量控制

锚点贪多会互相靠近、丧失区分度；如何在「覆盖足够的风格分布」与「锚点近正交、彼此可区分」之间取舍，
是否需要量化锚点间最小距离来约束新增。

### OQ4 — 场景判断从哪些信号推

「你在什么场景」需从 **repo / 语言 / git 习惯**推断（典型对立：**成熟大仓 + PR 工作流** vs **一次性脚本**）。
具体用哪些可派生信号、如何与 per-project 现状风格联动、跨平台一致性如何保证，待 E1/E4 落地后定。
