## 背景

PRD §3 要求：用脚本采集 Claude Code / Codex 的全局用量统计，把语义化结果交给大模型分析，
产出建议报告。落地前需要先定几个关键取舍。

## 决策

### D1 — 采集与分析解耦，digest 作为稳定中间产物

采用三段式：`collect → digest（语义化 JSON）→ advise（调用 LLM）`。
digest 是自带字段释义的结构化数据，作为采集与模型之间的契约。

- **理由**：沿用现有 `codexreport.Build()`（纯聚合）与 `Run()`（渲染/输出）分离的模式，
  让 digest 可在无模型环境下单测；模型只是 digest 的一个消费者。

### D2 — 用本机已登录的 CLI 做分析，不引入第三方网络依赖

分析调用走 `internal/provider`（已封装 `codex exec` / `claude -p`），
而非直接对接某个云端 API key。

- **理由**：用户已为这些 CLI 登录付费；不外发到额外服务符合 PRD 的隐私优先与离线友好；
  复用 provider 的超时与失败文案，减少新代码面。
- **代价**：受本机 CLI 的输出格式与稳定性约束；需要 `LookPath` 预检。

### D3 — 默认不外发 + 强制可预览（`--dry-run`）

发送给模型的内容必须可被用户预览；提供 `--dry-run` 只打印「将发送的内容」而不调用模型。

- **理由**：用量数据可能含仓库名、命令片段等敏感信息（digest 的来源里有 top 命令、仓库列表）。

### D4 — 分析对象与分析模型解耦

`--target`（分析谁的用量：codex/claude/all）与 `--provider`（用哪个模型来分析）是两个维度。

- **理由**：用户可能想「用 claude 点评 codex 的用量」或反之。

### D5 — 报告必须显式标注口径限制

模型输出里强制带上「仅本机数据 / 成本为估算 / 不含配额百分比」等免责口径，
并在 Prompt 中明确禁止模型编造配额百分比。

- **理由**：尊重 §2 既有边界，防止模型幻觉出账号级配额数字而误导用户。

## 待定（Open Questions）

- **OQ1 — 命令形态**：新增 `autofresh advise` 子命令，还是复用 `report --advise`？
  倾向新增子命令（语义清晰、参数互不污染），待实现时定。
- **OQ2 — Claude Code 数据源**：Codex 侧已有 rollout 解析；Claude Code 本地用量数据的
  位置与格式需先调研（见 TODO T1.1），结果可能反过来影响 digest 结构。
- **OQ3 — 大输入处理**：长时间窗口的 digest 可能很大，是否需要在发送前做摘要/截断分级。

## 后果

- 好处：复用现有聚合与 provider，新增面小；隐私与口径有明确护栏；易测试。
- 风险：依赖本机 CLI 的输出稳定性；Claude 侧数据源尚未确定，可能拖慢 T1。
