## 背景

现状：每次 `ccoach`（默认报告）都把 Claude Code 与 Codex **一起出**，产出一份双平台合并报告。落地后暴露三个问题：

1. **没回答任何新问题**。合并报告 = 两套指标并排呈现，用户得到的是「Claude Code 的一切 + Codex 的一切」，而不是
   「对照才能讲的洞察」。并排不等于对照。
2. **信息过载**。两份完整指标叠在一张报告里，关键信号被稀释，违背「教练价值在深度」的产品定位。
3. **约 2× 时间**。两个平台各跑一次解析（两次 CLI 查询量级），但绝大多数用户主力只在一个平台，第二次查询基本是
   浪费——而 CLI 默认路径**必须快**。

更严重的是**指标语义被污染**：Codex 与 Claude Code 的模型、harness 风格、交互范式完全不同。把两者合并算出的
「打断率 / 计划度 / 编码自主度」是**苹果加橘子**——例如 Codex 与 Claude Code 的「打断」语义不可通约，合并均值会
**污染人格判定**（ADR 0040 的风格/人格建模若吃合并指标，结论失真）。

但**对照本身是 ccoach 独有、别家做不了的洞察**——前提严格：**只在「只有对照才能讲」的内容上做对照**。把对照硬塞进
默认报告，既慢又乱还失真；把对照做成显式、独立、聚焦的报告类型，才把这份独有价值兑现出来。

本 ADR 依赖 E4 增量缓存（ADR 0035，per-platform profile）把「both 平台」的边际成本压到趋零。**E4 缓存在本切片不实现**，
此处仅留蓝图占位与接口预期；compare 在缓存就绪前可接受较慢。

## 决策

### D1 — 默认单平台深挖（草案）

默认报告**自动检测主力平台**，只解析、只深挖那一个：

- 检测口径：**最近 N 天活跃度占比**（草案 `N = 30`，以 session 数或 episode 数计，二选一在实现时定）。
  某平台占比 **≥ 阈值（草案 `≥ 80%`）** 即判为「单平台用户」，默认只走该平台。
- 行为：默认 `ccoach`（即 `report`）只跑主力平台的解析与深挖；第二平台**不触发第二次解析**，省掉 ~2× 时间。
- 理由：**教练价值在深度，CLI 默认路径必须快**。单平台用户拿到的是一份更深、更快、指标语义干净的报告。
- 兜底：若无明显主力（占比未过阈值，如 60/40 混用），按 OQ1 决定默认行为（候选：仍取占比第一名 / 提示用户显式选
  `--platform` 或跑 `compare`）。

字段层面预期新增 `--platform claude-code|codex`（显式覆盖自动检测）与内部 `primary_platform` 推断结果，写进报告头部
（让用户知道「这份报告只讲哪个平台、为什么」）。

### D2 — 风格/人格推断 per-platform 建模，不合并（草案）

风格漂移、人格判定（ADR 0040）**本就该 per-platform 建模**，绝不在合并指标上算：

- 同一用户在 Claude Code 与 Codex 上是**两套独立画像**，各自有自己的 `打断率 / 计划度 / 编码自主度` 基线。
- 与 ADR 0035 的 **per-platform profile** 对齐：profile 以「平台 × 画像」为粒度缓存，天然支持 per-platform 风格建模，
  也为 D3 的「风格漂移对照」提供两侧可比的归一化基线。
- 红线：任何「跨平台均值 / 合并人格」都不产出；要讲跨平台只能走 D3 的对照报告，且讲的是**差异**而非合并值。

### D3 — compare 做成显式独立报告类型（草案）

新增显式入口：CLI 侧 `ccoach compare`，skill 侧 `/ccoach-insight compare`。它**只讲对照专属内容**——
凡是单平台报告已能讲的，一律不在 compare 重复。专属内容三类：

1. **同类任务归一化成本**：基于 episode 抽象（ADR 0032）按 `task_type` 归一——「同样改 bug 类的 episode，
   Codex vs Claude Code 平均各烧多少」。归一化后成本才可比，避免拿不同任务结构对账。
2. **风格漂移**：同一用户跨平台的画像差异——「在 Claude Code 是规划官（高计划度），切到 Codex 变放养（低计划度）」。
   依赖 D2 的两侧 per-platform 画像作差。
3. **任务路由建议**：据前两项给可执行建议——「重构 / 长任务走 Claude Code，快速脚本走 Codex」。
   这是 compare **唯一独有的行动价值**：告诉用户**哪类活该去哪个平台**。

compare **不重复**单平台深挖内容；进入 compare 需满足最小数据要求（OQ2）。

### D4 — 速度分层：默认快、compare 慢可接受（草案）

- 默认报告（单平台）**慢不可接受**——是高频路径。
- compare（双平台）**慢一点可接受**——是显式、低频、按需触发的报告。
- 收敛路径：靠 E4 增量缓存（ADR 0035）把「both 平台」的边际成本**趋零**——首次跑两平台后，后续 compare 命中
  per-platform 缓存只算增量。E4 就绪前，compare 全量解析两平台、较慢，属可接受降级；**默认报告不依赖 E4 也必须快**
  （因为它只跑一个平台）。

## 后果

- **默认报告更快更深更干净**：单平台路径省掉 ~2× 解析时间，指标语义不再被跨平台合并污染，深度提升。
- **对照价值真正兑现**：compare 从「两套指标并排」升级为「只讲对照才能讲的洞察 + 任务路由建议」，成为 ccoach 的
  差异化招牌，且不拖慢高频默认路径。
- **风格/人格建模口径收紧**：ADR 0040 的人格判定吃 per-platform 指标，结论更可信；任何合并人格被明确禁止。
- **`README`「both platforms first-class and symmetric」承诺不破**：对称指**两平台都是一等公民、各自能被深挖、能进
  compare**——但「对称」**不等于「默认强行同框」**。需同步校准 README/文案措辞，把「默认双平台报告」改为
  「默认深挖主力平台 + 显式 `compare` 做跨平台对照」，避免读者误以为降级了多平台支持。
- **改动面（预期）**：CLI 新增 `--platform` 与 `compare` 子命令、`primary_platform` 推断（活跃度占比）、报告头部标注；
  skill 侧 `/ccoach-insight compare` 入口与对照渲染；依赖 ADR 0032 的 episode `task_type` 归一与 ADR 0035 的
  per-platform profile/缓存（**E4 本切片留占位**）。`--json` 契约需新增 compare 形态而**不破坏**现有单平台 schema。

## 开放问题

### OQ1 — 主力平台检测阈值与无主力兜底

`N 天`（草案 30）、占比阈值（草案 ≥80%）、以 session 还是 episode 计活跃度，均需用多份真实画像校准。无明显主力时
（如 60/40）默认行为待定：取占比第一名 / 提示显式 `--platform` / 自动建议跑 `compare`，三选一在实现时定。

### OQ2 — compare 最小数据要求

跨平台对照要有意义，两侧都需足量同类任务。需定**最小门槛**：每侧最少 episode 数、最少同 `task_type` 重叠数
（否则归一化成本/风格漂移样本不足、噪声大）。不达标时 compare 应**降级提示**（「Codex 数据不足以对照，先多用几次」）
而非硬出失真结论。

### OQ3 — 跨平台指标可通约性边界

哪些指标可归一化对照（成本按 task_type 归一是可通约的）、哪些**绝不可**（打断率/计划度的语义跨 harness 不可通约，
只能讲「同一用户在两平台各自的画像差异」而非「谁的打断率更高」）。需明确 compare 报告里**可对账的值** vs
**只可叙述差异的画像**的分界，防止把不可通约的指标做成数值排名。

### OQ4 — E4 缓存接口契约（蓝图占位）

D4 依赖 ADR 0035 的 per-platform 缓存把 both 边际成本趋零。E4 本切片不实现，但需提前约定接口：缓存以
「平台 × 画像」为键、compare 命中后只算增量。E4 落地前 compare 走全量解析（较慢、可接受）；E4 落地的具体 schema
与失效策略由 ADR 0035 定，本 ADR 仅声明依赖与降级路径。
