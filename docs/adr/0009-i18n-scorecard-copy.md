# ADR 0009 — 成绩卡固定文案的 i18n（人工本地化，非模型翻译）

> 状态：已接受（已实现） · 日期：2026-06-02 · 相关：[ADR 0008](0008-gamified-shareable-scorecard.md)、
> [`PRD.md`](../PRD.md) §3.11、[`TODO.md`](../TODO.md) T7 · 文案表：[`scorecard-copy.md`](../scorecard-copy.md)

## 背景

成绩卡里有两类文字，来源不同、处理方式也必须不同：

1. **模型即时生成**的内容——「人格总结」称号那段又准又毒的吐槽，喂用量数据让 agent 现写
   （[ADR 0008](0008-gamified-shareable-scorecard.md) D3）。这类天然随用户语言走，模型直接用对应语言写。
2. **预先写好的固定创意文案**——段位名（「玄学召唤师」「Opus 锤钉子」…）、每个段位的吐槽语、
   UI 标签（区块标题、「超过 X% 用户」等）。

第 2 类**不能靠模型即时翻译**：这些梗的精髓在于**本地化的语言味道**。「玄学召唤师」直译成
"Metaphysics Summoner" 就没味了；英文得**另起一个本地化的梗**（如 "Vibe Summoner" /
"Prompt Whisperer"）。逐字翻译会把段位从「好笑」变成「尴尬」，直接毁掉传播力。

## 决策

### D1 — 固定创意文案做成传统 i18n 文案表（人工打磨，每语言一套）

段位名 / 吐槽语 / UI 标签按语言各维护一套**人工本地化**文案，按用户语言选用对应那套；
**绝不**对这类文案做运行时机器/模型翻译。

### D2 — 首批支持中文 + 英文

`zh` 与 `en` 两套先行；表结构允许后续加语言。初版文案见 [`scorecard-copy.md`](../scorecard-copy.md)。

### D3 — 与模型生成内容分工明确

- 固定文案（段位名 / 吐槽语 / UI 标签）→ 走 i18n 文案表（本 ADR）。
- 称号 / 人格总结那段吐槽 → 模型按用户语言现写（ADR 0008 D3）。

### D4 — 语言选择

按用户/环境语言挑选文案集（具体探测方式实现时定，见 OQ）；缺失语言回退到 `en`。

### D5 — 本地化要尊重「损但不伤人」分寸

各语言的吐槽语各自满足 [ADR 0008](0008-gamified-shareable-scorecard.md) D5：调侃可改变的行为习惯，
不攻击能力 / 人格。本地化不是直译，但红线一致。

## 待定（Open Questions）

- **OQ1**：语言探测来源（系统 locale / agent 语言 / 用户显式指定 / skill 参数）。
- **OQ2**：文案表的承载形式（skill 内 JSON/YAML 资源，供渲染脚本读取）。
- **OQ3**：未来加语言时的协作流程（谁来本地化、如何评审梗的「味道」）。

## 后果

- 好处：每种语言的段位梗都「地道」，保住传播力；与模型生成内容职责清晰。
- 代价：每加一种语言都要人工打磨一套文案（不能自动翻），并随段位调整同步维护。
- 影响：渲染层按语言查文案表；新增 [`scorecard-copy.md`](../scorecard-copy.md) 作为初版 zh/en 文案，
  实现时落成 skill 内的 i18n 资源。
