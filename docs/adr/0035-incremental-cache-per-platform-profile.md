# ADR 0035 — 增量缓存 + per-platform profile（`~/.ccoach`）

> 状态：提议中（pending） · 日期：2026-06-05
> · 底座承接 [`adr/0032-episode-abstraction-layer.md`](0032-episode-abstraction-layer.md) 的 E1 episode 层（按 episode 持久化、增量解析）
> · 复用 [`adr/0018-cli-absorbs-collection-prompt-preview.md`](0018-cli-absorbs-collection-prompt-preview.md) 的采集/解析口径（一个 pass 出「用量 + prompt + 习惯」）
> · 沿用 [`adr/0017-derived-non-content-signals.md`](0017-derived-non-content-signals.md) 的「派生信号、零原文、瞬时读即弃」红线
> · 喂给 [`adr/0036-quantified-counterfactual-baselines.md`](0036-quantified-counterfactual-baselines.md) 的量化反事实基线（profile 提供个人化基线）
> · 喂给 [`adr/0040-style-space-archetype-personas.md`](0040-style-space-archetype-personas.md) 的风格空间/原型人格（profile 存风格向量与目标风格）

## 背景

当前双平台查询（`both`）约比单平台慢 2x，根因不是 I/O 也不是算法复杂度，而是**每次都对两套 JSONL 做全量重复解析**。
Claude Code / Codex 的会话日志是 **append-only**（只追加、历史行不变），这是缓存友好到天赐的结构：已解析过的文件，
除非 `mtime` 变化，否则其历史派生结果可以原样复用。

更深一层是**教练黄金法则「数据能测的绝不问」**（呼应 ADR 0020 自述教练语气）。一个像样的教练不会每次见面都从零问
「你最近想冲速度还是稳？」——他记得上次的结论，把**用户画像持久化**，下次直接拿来用；只有当**数据本身显示风格漂移**时，
才回头确认一次。要做到这点，光有「每次重算」的无状态 CLI 是不够的，需要一层**跨调用持久化**。

本 ADR 是 ADR 0032 的 **E1 episode 抽象层之上的持久化底座**：episode 给出「按 episode 聚合」的稳定单元，本 ADR 给出
「把这些单元缓存下来、把用户画像存下来」的机制。**本切片不实现，仅落蓝图、占编号位**——先把数据结构、隐私边界、
失效策略想清楚，避免后续 ADR 0036 / 0040 各自发明一套临时缓存。

## 决策

> 以下 D1–D4 均为**草案**（pending），描述目标形态与约束，不在本切片落地。

### D1（草案）增量缓存：`~/.ccoach/cache/`，episode 级按日聚合

- 缓存落在 `~/.ccoach/cache/`，以 **episode 级、按日聚合**为粒度存派生结果（与 ADR 0032 的 E1 单元同构）。
- **首次全量解析**后建立缓存；后续运行只**增量解析 `mtime` 变化的文件与其中新增的行**，已解析的历史行直接读缓存。
- 目标：把 `both`（Claude Code + Codex 双平台）的**边际成本趋近于零**——稳态下只解析当天还在写的活跃日志。
- 缓存键需含**文件路径 + `mtime` + 已解析到的偏移/行游标**，以便判定「哪些行是新增的」。
- 缓存命中与否对 `--json` 契约**完全透明**：输出字段、语义不变，skill 侧无感（不破坏 ADR 0004 / 0010 的契约）。

### D2（草案）per-platform profile：`~/.ccoach/profile.json`，按平台分开存

- 持久化用户画像到 `~/.ccoach/profile.json`，至少含：**风格向量**（喂 ADR 0040）、**目标风格**（用户声明的取向，如「冲速度」）、
  **置信度**、**`last_confirmed_at`**（上次与用户确认画像的时间戳）。
- profile **天然 per-platform**：Codex 与 Claude Code 的 plan mode / 记忆 / 打断语义 / auto 模式**非同构**，
  把两边指标合并算一个总画像是「苹果加橘子」。因此 profile 按平台分桶存储，**不跨平台合并出单一画像**。
- profile 为 ADR 0036 的量化反事实基线提供**个人化基线**（「相对于你自己过去三周」而非相对于人群均值）。

### D3（草案）漂移检测：指标偏离已存 profile 阈值 → 提示重新确认

- 每次运行把当前窗口的风格指标与已存 profile 比对；**偏离超过阈值**即判定为「风格漂移」，触发一次**重新确认**提示。
- 提示文案本身就是高级教练时刻，例如：「三周前你说要冲速度，但最近打断率翻倍，目标变了吗？」
  （此处「打断率」等只用 ADR 0017 口径的派生计数，不读任何原文。）
- 确认是**低频、按需**的：未漂移则沉默直读 profile（落实「数据能测的绝不问」）；确认后刷新 profile 与 `last_confirmed_at`。
- 阈值需 per-platform、per-指标设定（不同信号的自然波动幅度不同），具体取值留待实现期标定。

### D4（草案）隐私：缓存只存派生信号、零原文、不跨机器

- 缓存与 profile **只存派生信号**，口径与 ADR 0032 的 `episodes_detail` / ADR 0017 完全一致：
  布尔、纯计数、固定白名单标签、风格向量等聚合量，**零 prompt 原文、零 assistant/thinking/system 内容、零文件/命令全行**。
- 派生过程沿用「瞬时读 → 派生 → 即弃」红线：原始 stderr/stdout/diff 只在派生那一刻用于算计数，**绝不写进缓存文件**。
- 缓存**只反映本机**，**绝不跨机器汇总**（profile 是账号级取向的本机投影，不上传、不合并多机）。
- `~/.ccoach/` 全程本地、默认不外发；可分享成绩卡仍走纯聚合、零原文路径（不受本缓存影响）。

## 后果

- **正面**：`both` 稳态查询从「每次全量 2x」降到「近增量、近零边际成本」；CLI 从无状态升级为有「记忆」的教练底座，
  让 ADR 0036（个人化基线）/ ADR 0040（风格原型）有稳定、可复用的数据源，无需各自造缓存。
- **正面**：「数据能测的绝不问」从口号变成机制——画像直读、低频确认，交互更像真教练。
- **代价**：引入磁盘状态即引入一类新故障面（缓存与真相不一致、损坏、schema 漂移），需要严格的失效/回退策略（见 OQ）。
- **代价**：per-platform 分桶意味着 profile / 缓存的存储与比对逻辑要按平台展开，代码与测试量上升。
- **约束不变**：`--json` 契约、隐私红线、本机-only 全部不放宽；缓存是**纯加速 + 持久化**层，不改变任何对外语义。
- **范围**：本 ADR 不写代码，仅作蓝图；落地时另起实现切片并补回归（见 OQ3）。

## 开放问题

### OQ1 — 缓存失效与版本迁移
缓存如何失效？解析器逻辑升级（派生口径变了）后，旧缓存必须整体失效或迁移。考虑给缓存打**解析器版本号 / schema 版本号**，
版本不匹配即重建。是否需要 `ccoach --no-cache` / `ccoach cache clear` 之类的逃生口？

### OQ2 — profile schema 版本化
`profile.json` 的字段（风格向量维度、目标风格枚举）会随 ADR 0040 演进。需要 schema 版本字段 + 前向/后向兼容迁移策略；
旧版本 profile 读不动时，是丢弃重建还是尽力迁移并保留 `last_confirmed_at`？

### OQ3 — 增量正确性回归
必须保证**增量解析结果与全量解析逐字段一致**。需要一条回归测试：同一组 fixture，分别走「全量」与「先全量建缓存再增量」，
断言两条路径产出的统一结构完全相同（接入 vitest，类比 `scripts/verify-ccusage.ts` 的对账思路）。

### OQ4 — 缓存损坏时安全回退全量
缓存文件被截断 / 写坏 / 手改坏时，必须**安全回退到全量解析**而非崩溃或出错数据。读缓存需做校验（如校验和 / 结构校验），
失败即静默丢弃该缓存重算。回退要可观测（debug 日志），但不打断用户。

### OQ5 — episode 边界与缓存粒度的耦合
缓存按 episode 级按日聚合，依赖 ADR 0032 的 episode 切分稳定。若 episode 边界算法后续调整，是否会使旧缓存的聚合单元失配？
需明确「episode 切分版本」也纳入缓存失效键。
